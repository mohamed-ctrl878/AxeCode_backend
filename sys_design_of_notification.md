هذا ملف تصميم نظام (System Design Document) شامل لخدمة تنبيهات عامة (Generic Notification Service) تركز على المتصفح (Web Push) ومصممة لتكون قابلة للتوسع والربط مع أي خدمة أخرى.System Design: Generic Notification Service (Web-First)1. نظرة عامة (High-Level Overview)الهدف هو بناء خدمة مركزية تستقبل طلبات التنبيه من مختلف الخدمات (Microservices) وتوصيلها للمستخدمين عبر المتصفح، مع إمكانية إضافة قنوات أخرى مستقبلاً (Email, SMS).2. المكونات الأساسية (System Components)أ. Notification Producer (The Placeholder)هذا هو الجزء الذي سألته عنه؛ أي خدمة داخل نظامك تريد إرسال تنبيه (مثل خدمة الطلبات، خدمة المحفظة، إلخ).الوظيفة: ترسل حدثاً (Event) لخدمة التنبيهات.المحتوى (Payload Placeholder):JSON{
  "target_user_id": "UUID",
  "event_type": "ORDER_UPDATE", // نوع الحدث لتحديد القالب
  "content_data": { 
     // هنا الـ Placeholder للمحتوى المتغير
     "title_param": "قيمة متغيرة",
     "body_param": "قيمة متغيرة",
     "action_url": "/orders/123"
  },
  "priority": "HIGH"
}
ب. API Gateway & Authenticationتستقبل الطلبات من الـ Producers وتتأكد من صلاحيتهم لإرسال التنبيهات.ج. Notification Worker (The Brain)المحرك الرئيسي الذي يقوم بـ:Template Engine: دمج بيانات الـ content_data مع القالب المخزن (مثلاً: "عزيزي {{name}}، طلبك رقم {{id}} جاهز").User Preferences: التحقق مما إذا كان المستخدم مفعلاً لإشعارات المتصفح أم لا.Device Registry: جلب الـ Subscription Tokens الخاصة بمتصفحات المستخدم من قاعدة البيانات.د. Web Push Provider (FCM/VAPID)الوسيط الذي يرسل الإشعار الفعلي للمتصفح (مثل Firebase Cloud Messaging).3. تصميم قاعدة البيانات (Data Schema)لضمان أن الخدمة Generic، نحتاج للجداول التالية:Templates Table: لتخزين نصوص التنبيهات بلغات مختلفة.Subscriptions Table: (الأهم للمتصفح)user_id, endpoint, p256dh_key, auth_key, browser_type.Notification Logs: لتتبع حالة التنبيه (Sent, Delivered, Clicked).4. مسار البيانات (Data Flow)الخدمة الخارجية (Placeholder): ترسل طلباً يحتوي على user_id وبيانات الحدث.Queue: يتم وضع الطلب في صف (Message Queue مثل Redis) لضمان عدم ضياع التنبيه ولتخفيف الضغط.Processor: يقوم الـ Worker بسحب الطلب، وجلب الـ Web Push Tokens الخاصة بالمتصفح المسجل للمستخدم.Delivery: يتم تشفير المحتوى باستخدام مفاتيح الـ VAPID وإرساله لمزود الخدمة (Push Service).Service Worker: يستقبل المتصفح الإشارة، ويقوم الـ Service Worker بعرض الإشعار حتى لو كان الموقع مغلقاً.5. التعامل مع محتوى التنبيه (Extensible Content)لجعل المحتوى مرناً (Generic)، نستخدم استراتيجية Dynamic Payload:الحقلالدور في الخدمةTemplate_IDيحدد شكل الرسالة (مثلاً: قالب الترحيب، قالب التنبيه المالي).Placeholdersقاموس (Dictionary) يحتوي على القيم التي ستُحقن في القالب.Deep Linkرابط يوجه المستخدم لصفحة معينة في المتصفح عند الضغط على الإشعار.6. اعتبارات التوسع والأمان (Scalability & Security)VAPID Keys: يتم توليد زوج مفاتيح واحد للسيرفر بالكامل لتوثيق الإرسال.Retry Mechanism: في حال كان متصفح المستخدم غير متاح لحظياً، يتم إعادة المحاولة باستخدام Exponential Backoff.Deduplication: لمنع إرسال نفس الإشعار مرتين للمستخدم في حال تكرار ضغط الزر في الخدمة المرسلة.Batching: إرسال التنبيهات لمجموعة كبيرة من المستخدمين في دفعات (Batches) لتقليل استهلاك الموارد.7. مخطط الرحلة (Sequence Diagram)User يفتح المتصفح -> يتم تخزين الـ Subscription في قاعدة بيانات التنبيهات.Feature Service -> تطلق حدثاً لخدمة التنبيهات.Notification Service -> تحول الحدث إلى رسالة منسقة.Push Service (Google/Mozilla) -> تستقبل الرسالة المشفرة.User Browser -> يعرض الإشعار للمستخدم.
